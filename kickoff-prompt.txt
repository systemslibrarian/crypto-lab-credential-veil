ADDITIONAL BINDING CONSTRAINTS FOR THIS LAB (this is a "system fails, primitive holds"
lab — these are non-negotiable and outrank feature completeness):

  A. VERDICT SEPARATION. Cryptographic result and security verdict are SEPARATE,
     independently-rendered indicators, never collapsed. "Signature: valid ✓" next to
     "Verdict: REJECT ✗" is the point of the lab. Per §1 VISUAL SEMANTICS, color tracks
     system integrity, never the raw return value: a forged-but-accepted result renders
     as ALARM, not green.
  B. NO PRECISION DRIFT. Every consequence claim must be exactly true for the exact
     construction. If the brief distinguishes authentication-key recovery from
     encryption-key recovery, that distinction must survive into the UI copy verbatim.
     A slightly-overstated attack claim is a §0.2 honesty violation and fails the build.
  C. SCOPE GUARD. The Non-goals list is binding. If a stage is interesting but doesn't
     serve the "Concept to teach", cut it and add a one-line "what this isn't" note per
     §1 SCOPE. When something on the cut list is genuinely adjacent, link to the sibling
     demo — do not rebuild it.
  D. THE LEARNER CAUSES THE FAILURE. The break-it-yourself interaction must run against
     the real primitive and the real verifier. No warning banner substitutes for a button
     that makes genuine crypto produce a genuinely wrong outcome.
